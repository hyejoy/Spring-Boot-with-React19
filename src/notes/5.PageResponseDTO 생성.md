# Spring Boot: 수정 · 삭제 · 목록 처리 & PageRequestDTO

## 수정 (modify)

### 왜 복잡한가?
수정 가능한 필드와 불가능한 필드가 섞여 있기 때문.

```
tno    → 자동 생성 PK, 변경 불가
writer → 작성자, 변경 불가
title  → 변경 가능 ✅
complete → 변경 가능 ✅
dueDate  → 변경 가능 ✅
```

### 수정 흐름

```
1. DB에서 Entity 조회 (findById)
2. change() 메서드로 변경 가능한 필드만 수정
3. save() 또는 Dirty Checking으로 자동 반영
```

```java
@Override
public void modify(TodoDTO todoDTO) {
    Optional<Todo> result = todoRepository.findById(todoDTO.getTno());
    Todo todo = result.orElseThrow();

    // 변경 가능한 필드만 수정 (tno, writer는 그대로 유지)
    todo.change(todoDTO.getTitle(), todoDTO.isComplete(), todoDTO.getDueDate());

    todoRepository.save(todo);
}
```

## Dirty Checking (변경 감지)

JPA의 기본 기능. `@Transactional` 안에서 Entity가 변경되면 **save() 없이도 자동으로 UPDATE 쿼리 실행**.

```java
// @Transactional 환경에서
todo.change(title, complete, dueDate);
// → 트랜잭션 종료 시 JPA가 변경 감지 → UPDATE 자동 실행
```

> 테스트 코드에서는 `@Transactional`을 직접 걸어줘야 Dirty Checking 동작

## 삭제 (remove)

### 실제 삭제(Hard Delete)는 하지 않는다

```java
// ❌ Hard Delete — 실무에서 사용 금지
todoRepository.deleteById(tno);

// ✅ Soft Delete — boolean 값으로 삭제 여부 표시
todo.change(..., true, ...); // complete = true 처럼 삭제 플래그 활용
```

> "해당 게시물은 삭제되었습니다" 처리 방식이 바로 Soft Delete

### 반환 타입을 void로 설계하는 이유

```
수정/삭제 실패 → 예외(Exception)로 처리
→ 리턴값보다 예외를 던지는 게 객체지향적으로 올바른 설계
```

## 목록 처리 — PageRequestDTO / PageResponseDTO

### 왜 DTO로 묶나?

목록 조회 시 파라미터가 너무 많아짐:

```
page, size, sort, keyword, type, startDate, endDate ...

// ❌ 파라미터 나열 방식
getList(int page, int size, String keyword, String type, LocalDate start ...)

// ✅ DTO로 묶기
getList(PageRequestDTO pageRequestDTO)
```

> 2개 이상의 데이터를 던질 때는 DTO로 묶는 게 유지보수에 유리

### PageRequestDTO

```java
package com.example.dto;

import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Data;
import lombok.NoArgsConstructor;
import lombok.experimental.SuperBuilder;

@Data
@SuperBuilder  // 상속해서 검색 조건 추가 가능하도록
@AllArgsConstructor
@NoArgsConstructor
public class PageRequestDTO {

    @Builder.Default
    private int page = 1;   // 기본값: 1페이지

    @Builder.Default
    private int size = 10;  // 기본값: 10개
}
```

### SuperBuilder를 쓰는 이유

```
PageRequestDTO (공통: page, size)
    ↑ 상속
TodoSearchRequestDTO (추가: keyword, type, startDate ...)
BoardSearchRequestDTO (추가: category, keyword ...)
```

검색 조건은 화면마다 다름 → 공통 부분은 부모 DTO, 추가 조건은 자식 DTO에서 확장

### @Builder.Default 를 쓰는 이유

```java
// Builder 패턴 사용 시 기본값이 무시되는 문제 발생
// @Builder.Default 를 붙여야 기본값이 정상 적용됨
@Builder.Default
private int page = 1;
```
